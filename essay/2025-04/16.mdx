这是我了解完且上手过 MCP 之后，参加大佬的技术沙龙引发的思考。先聊一下 MCP 和 Function Calling 的区别。

你把模型想象成一个医生，目前他只能凭借自己的认知和经验来帮病人看病，根据用户的一些症状表征进行诊断。 \
当有了 Function Calling 能力之后，就像是医生可以开检查单和化验单，让病人去做一些相关的检查，借助其他科室的能力来帮助诊断。
MCP Client 应该是带着检查单和化验单的用户，用户将检查单和化验单送到对应科室进行检查，拿到检查结果给医生，让医生做出更准确的诊断。 \
MCP Server 就是其他的科室，它们可以提供一些专业的检查和化验服务，最后将检查结果给到病人，让病人给医生看。 \
但是对于大型医院，通常是医院内直接提供检查的地方，不用用户自己去外面找。这种 MCP Server 就是单点的，好处是直接在医院内部不用到处去找检查的地方。 \
弊端也很明显，人多就会排队，等待时间长，体验差。这时候对于一些常见的病情的检查手段也被提供了，允许用户直接采买相关的器械，自己在家进行检查。将检查结果反馈给医生。 \
故而 Function Calling 对模型而言，就像开检查和化验单对医生来说一样只是一种能力，并且医生并不会真正的给病人进行检查和化验，而是让病人将检查单和化验单交给病人带到其他科室，让其他科室进行检查和化验，医生只做最后的检查结果的诊断。

1. 模型不会执行 Function Calling 定义的函数，真正执行的是用户的机器
2. MCP 分为 Client 和 Server，Client 本质上是用户（这个场景下，只有用户能传递检查单和化验单），而 Server 就是检查或化验的科室，他们根据化验单和检查单的内容（Schema）来为用户提供服务

:::info
MCP 肯定会成为模型调用工具的标准，但是 MCP 的也存在很大的弊端，当接入的 MCP Server 太多了，模型需要加载的 Schema 也越多，整体的上下文占据的空间也会越大。 
:::

现在聊一下我在工程内遇到的一些问题：Agent 调用 MCP Server 的鉴权问题，我当时在做 IDaaS Agent 的时候遇到了一个世纪难题，我应该如何去设计 MCP Server 的权限控制才能让 Agent 能够合理的去使用？
场景其实就是 IDaaS 管理控制台那套逻辑，对一些资源的增删改查，因为 IDaaS 是底层组件并且接入具有很高的成本（概念比较复杂），故而需要这样一个 Agent 去代替用户做管理的事情。
最开始我聚焦于设计 MCP Server Metadata，让每个 tool 的 metadata 内都包含需要申请的 scope，在 Agent Loop 内设计一环工具调用的鉴权层并结合 HITL 统一向用户申请当次执行操作的权限（仅一次）。
这是将用户的体验优化到了极致，但是这并不是最佳实践，参考 anthropic 和 cursor，他们只有在需要执行某些工具的时候才会向用户申请权限，所以会多次询问用户是否授权，用户理应关心每次执行的操作。当然也允许用户直接一次性完全授权，但是肯定会做出提示。
所以后续我调整完成后结合飞书的能力（通过机器人 + 带按钮的卡片来告诉用户并且引导用户授权），我完成这套设计的落地。当然这只是一种我想出来的解决方案，针对 MCP Server 鉴权的解决方案。
